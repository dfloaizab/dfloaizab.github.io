1. Características y Estructura del Proceso

Imagen del Proceso (Core Image)

Un proceso es una instancia de un programa en ejecución que reside en un espacio de direcciones [1, 2]. La Imagen del Proceso es la colección completa de elementos necesarios para su ejecución: el código del programa (segmento de texto), los datos (globales y estáticos) y la pila (stack) para llamadas a funciones [2, 3]. En sistemas modernos, esta imagen puede estar parcialmente en memoria o en disco mediante paginación [4, 5].

El PCB (Bloque de Control de Proceso)

El PCB es la estructura de datos más importante para el SO, permitiendo la multiprogramación [6, 7]. Contiene Identificadores (PID, PID del padre), Información de Estado del Procesador (registros, contador de programa, punteros de pila) e Información de Control (estado del proceso, prioridad, parámetros de planificación y recursos asignados) [8-10]. El PCB es el "descriptor" que permite al kernel suspender un proceso y reanudarlo exactamente en el mismo estado [6, 11].

💬 2. Inter-Process Communication (IPC) y Sincronización

La IPC aborda tres problemas: paso de información, evitar que los procesos interfieran entre sí al acceder a datos compartidos y gestionar dependencias de secuenciación [12].

Primitivas de Sincronización

  • Semáforos: Variables enteras con operaciones atómicas Up (incrementar) y Down (decrementar/bloquear) [13, 14]. Un semáforo binario (mutex) garantiza exclusión mutua, mientras que los contadores gestionan recursos múltiples [15, 16].
  • Monitores: Construcción de lenguaje de alto nivel donde solo un proceso puede estar activo a la vez [17, 18]. Utilizan Variables de Condición con operaciones Wait y Signal para bloquear procesos hasta que se cumpla un predicado [19, 20].
  • Message Passing: Útil en sistemas distribuidos; usa primitivas Send y Receive [21, 22]. Puede ser bloqueante (sincrónico) o no bloqueante (asincrónico) [23, 24].
  • Barreras: Sincronizan grupos de procesos en fases; ninguno avanza hasta que todos lleguen a la barrera [25, 26].

Anomalías de Acceso

Las Condiciones de Carrera ocurren cuando el resultado final depende del orden preciso de ejecución [27, 28]. Para evitarlas, se definen Regiones Críticas, donde el acceso debe ser mutuamente exclusivo [29, 30].

📅 3. Planificación de Procesos (Scheduling)

El planificador busca optimizar criterios como la equidad, el rendimiento (throughput) y el tiempo de respuesta [31, 32].

Niveles de Planificación

  • Largo Plazo: Decide qué trabajos se admiten al sistema como procesos [33].
  • Medio Plazo: Gestiona el intercambio (swapping) entre memoria y disco [34].
  • Corto Plazo (Dispatcher): Selecciona qué proceso listo se ejecuta en la CPU a continuación [34].

Algoritmos Notables

  • Round Robin (RR): Asigna un quantum de tiempo fijo; si expira, el proceso se preempa [35, 36].
  • Shortest Process Next (SPN): Favorece tareas cortas, minimizando el tiempo de espera promedio [37, 38].
  • Multilevel Feedback Queues (FB): Penaliza a procesos que usan mucha CPU moviéndolos a colas de menor prioridad [39, 40].

🐧 El Caso de Linux

Linux usa el Completely Fair Scheduler (CFS) para tareas normales [41]. No usa colas de prioridad tradicionales, sino un Árbol Rojo-Negro para rastrear el tiempo de ejecución virtual (vruntime) de cada tarea, seleccionando siempre la que menos CPU ha recibido para maximizar la equidad [41, 42].

🔒 4. Gestión Exhaustiva de Interbloqueos

Un interbloqueo ocurre cuando un conjunto de procesos se bloquea permanentemente porque cada uno espera un evento que solo otro proceso del mismo conjunto puede causar [43, 44].

Las 4 Condiciones de Coffman (Necesarias)

  1. Exclusión Mutua: Los recursos se asignan a un solo proceso a la vez [45, 46].
  2. Retener y Esperar: Procesos con recursos pueden solicitar nuevos [45, 46].
  3. Sin Preopción: Los recursos no pueden quitarse por la fuerza [45, 46].
  4. Espera Circular: Cadena cerrada donde P1 espera a P2, P2 a P3... y Pn a P1 [47, 48].

Estrategias de Gestión

Prevención (Prevention)

Ataca estructuralmente una de las condiciones de Coffman, por ejemplo, forzando un orden numérico en la solicitud de recursos para evitar ciclos [49, 50].

Evitación (Avoidance)

Usa el Algoritmo del Banquero para decidir dinámicamente si otorgar un recurso es seguro (existe una secuencia de finalización garantizada) o inseguro [51, 52].

Detección y Recuperación

Permite que ocurran interbloqueos, los detecta buscando ciclos en grafos de asignación y recupera mediante la terminación de procesos o la preopción de recursos [53-55].
