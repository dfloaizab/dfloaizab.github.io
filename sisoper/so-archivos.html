01 · Sistemas de Archivos

Sistemas de Archivos

Un sistema de archivos es la capa del SO que organiza, almacena y recupera datos en dispositivos de almacenamiento. Abstrae los sectores físicos del disco en archivos y directorios con nombres y permisos.

Arquitectura en capas

Aplicaciónfopen(), read(), write() — API de alto nivel en C/POSIX
↕
Syscall Interfaceopen(2), read(2), write(2), close(2) — llamadas al kernel
↕
VFS (Virtual FS)Capa de abstracción: unifica ext4, NTFS, FAT32, procfs...
↕
Driver del FSImplementación específica (ext4, FAT32, ZFS…) — traduce VFS a bloques
↕
HardwareControlador de disco, SSD, NVMe — transferencias en bloques físicos

Estructuras internas clave

📄

Inodo (inode)

Estructura metadato de un archivo: tamaño, permisos, timestamps, propietario, y punteros a bloques de datos. NO contiene el nombre del archivo.

📁

Directorio

Tabla de {nombre → número de inodo}. El nombre vive en el directorio; el contenido y metadatos en el inodo. Permite hard links: dos nombres apuntando al mismo inodo.

🗂️

Superbloque

Metadatos del sistema de archivos completo: tamaño total, número de inodos libres, bloques libres, magic number para identificar el tipo de FS.

🗺️

Bitmap de bloques

Mapa de bits donde 1 = bloque ocupado, 0 = libre. Permite al FS encontrar espacio libre eficientemente sin escanear todo el disco.

🔗

FAT (File Alloc. Table)

En FAT32: tabla de cadenas de clústeres. Cada entrada apunta al siguiente clúster del archivo o marca EOF. Simple pero frágil (punto único de fallo).

📋

Journaling

Log (journal) de transacciones pendientes. Si el sistema falla a mitad de una escritura, el journal permite recuperar la consistencia. Usado en ext4, NTFS, XFS.

02 · Operaciones

Operaciones sobre Archivos

El SO provee a los procesos un conjunto de llamadas al sistema (syscalls) para manipular archivos. Internamente, cada syscall transita por el VFS hacia el driver específico del FS.

Syscall (POSIX)OperaciónDescripciónRetorna
open(path, flags)AbrirCrea una entrada en la tabla de archivos del proceso. Asigna un descriptor (fd). flags: O_RDONLY, O_WRONLY, O_CREAT, O_TRUNC, O_APPENDfd ≥ 0 o -1
read(fd, buf, n)LeerTransfiere hasta n bytes del archivo al buffer. Mueve el puntero del archivo. Puede bloquearse si no hay datos disponibles.bytes leídos
write(fd, buf, n)EscribirTransfiere n bytes del buffer al archivo. Puede ir al page cache antes del disco (escritura diferida). Actualiza mtime.bytes escritos
lseek(fd, offset, whence)PosicionarMueve el puntero del archivo. whence: SEEK_SET (absoluto), SEEK_CUR (relativo), SEEK_END (desde el final).posición nueva
close(fd)CerrarLibera el descriptor. Si nlinks=0 y ningún proceso lo tiene abierto, libera el inodo y los bloques de datos.0 o -1
stat(path, &sb)MetadatosLlena struct stat con: inodo, modo, uid, gid, tamaño, timestamps, nlinks. No abre el archivo.0 o -1
unlink(path)EliminarElimina la entrada del directorio (decrementa nlinks). El inodo/datos se liberan cuando nlinks llega a 0 Y no hay fds abiertos.0 o -1
mmap(addr, len, prot, flags, fd, off)Mapear en memoriaMapea un archivo en el espacio de direcciones del proceso. Acceso por puntero, sin read/write explícitos. La MMU gestiona la E/S.ptr o MAP_FAILED
fsync(fd)SincronizarFuerza la escritura de todos los datos y metadatos del archivo al dispositivo físico. Crucial para consistencia en bases de datos.0 o -1

Ejemplo en C — syscalls de archivos

Operaciones básicas con archivosC / POSIX
#include <fcntl.h>
#include <unistd.h>
#include <sys/stat.h>

int main() {
    /* 1. Abrir (crea si no existe, trunca si existe) */
    int fd = open("datos.bin", O_RDWR | O_CREAT | O_TRUNC, 0644);
    /* permisos 0644 = rw-r--r--                          */

    /* 2. Escribir */
    char buf[] = "Hola SO";
    write(fd, buf, sizeof(buf));

    /* 3. Reposicionar al inicio */
    lseek(fd, 0, SEEK_SET);

    /* 4. Leer de vuelta */
    char lectura[16];
    ssize_t n = read(fd, lectura, 16);

    /* 5. Obtener metadatos */
    struct stat st;
    fstat(fd, &st);
    /* st.st_size, st.st_mtime, st.st_mode … */

    /* 6. Sincronizar con disco (¡crítico para BD!) */
    fsync(fd);

    /* 7. Cerrar */
    close(fd);

    /* 8. Mapear en memoria (alternativa a read/write) */
    fd = open("datos.bin", O_RDONLY);
    void* ptr = mmap(NULL, st.st_size, PROT_READ,
                      MAP_PRIVATE, fd, 0);
    /* acceder ptr[i] como si fuera un array en RAM */
    munmap(ptr, st.st_size);
    close(fd);
}
03 · Interrupciones

Interrupciones de Hardware

Una interrupción es una señal asíncrona generada por hardware que fuerza a la CPU a suspender su ejecución actual, guardar su estado, y ejecutar una rutina de manejo de interrupción (ISR/IRQ handler) en el kernel.

Flujo completo de una interrupción

1
Dispositivo señaliza
El disco/teclado/NIC levanta la línea IRQ del bus
2
PIC / APIC captura
El controlador de interrupciones asigna un vector y notifica a la CPU
3
CPU finaliza instrucción
Completa la instrucción en curso (no es inmediato)
4
Contexto salvado
PC, registros, flags → pila del kernel (modo kernel)
5
ISR ejecuta
Kernel lee/escribe el dispositivo, actualiza buffers, despierta procesos
6
Contexto restaurado
IRET restaura registros y retoma el proceso interrumpido

Tipos de interrupciones

⚡

IRQ de hardware

Generadas por dispositivos externos: teclado (IRQ 1), disco IDE (IRQ 14), NIC (IRQ 11). Asíncronas — llegan en cualquier momento. Manejadas por el PIC/APIC.

⏱️

Timer / Reloj (IRQ 0)

La interrupción más importante del SO. Dispara cada 1–10 ms. El scheduler la usa para implementar time-slicing y expulsar procesos que agotan su quantum.

💾

DMA completado

El controlador DMA notifica que terminó la transferencia entre memoria y dispositivo. Permite que la CPU haga otro trabajo mientras el disco transfiere datos.

❌

Excepción / Fault

Generada por la CPU ante errores: división por cero (INT 0), fallo de página (INT 14), acceso a memoria ilegal. Pueden ser recuperables (page fault normal) o no (SIGSEGV).

🔢

NMI (Non-Maskable)

No puede ser deshabilitada. Reservada para fallas críticas de hardware: errores de paridad en RAM, fallos de watchdog. El SO sólo puede loggear y, generalmente, hacer panic.

📡

IPI (Inter-Processor)

En sistemas multiprocesador: una CPU interrumpe a otra para sincronización (TLB shootdown, migración de proceso, apagado de núcleo).

04 · Traps y Llamadas al Sistema

Traps y Syscalls

Un trap (o interrupción software) es una interrupción generada deliberadamente por el proceso para solicitar servicios del kernel — es el mecanismo fundamental que implementa las llamadas al sistema.

Diferencia: Interrupción vs. Trap vs. Excepción

TrapCPU (instrucción INT)✅ Síncrono✅ Sí (intencional)syscall: int 0x80, syscall
Excepción / FaultCPU (error)✅ Síncrono❌ No (error)División por cero, page fault, instrucción inválida
AbortCPU / HW crítico✅ Síncrono❌ NoError de paridad en RAM, fallo de doble fault

Ejemplo de syscall directa en ensamblador

sys_write desde Assembly x86-64NASM / Linux
; Escribe "Hola SO\n" en stdout usando syscall directa
section .data
    msg  db  "Hola SO", 10   ; 10 = '\n'
    mlen equ $ - msg

section .text
    global _start

_start:
    ; syscall write(fd=1, buf=msg, count=mlen)
    mov rax, 1          ; número syscall: write
    mov rdi, 1          ; fd: stdout
    mov rsi, msg        ; dirección del mensaje
    mov rdx, mlen       ; longitud
    syscall             ; ← TRAP → kernel
    ; al retornar, rax = bytes escritos (o -errno)

    ; syscall exit(0)
    mov rax, 60         ; número syscall: exit
    xor rdi, rdi        ; código de salida = 0
    syscall             ; proceso termina
05 · Periféricos

Estructura de Periféricos y Drivers

Un periférico es cualquier dispositivo de E/S conectado a la CPU: disco, teclado, NIC, pantalla, USB. El SO se comunica con ellos a través de drivers que hablan con los controladores hardware mediante registros mapeados en memoria o puertos de E/S.

06 · Comunicación CPU–Periférico

Modelos de Entrada/Salida

La forma en que la CPU coordina las transferencias con los periféricos es crítica para el rendimiento del sistema. Existen cuatro modelos fundamentales.

🔄

1 · E/S programada (Polling / Busy-wait)

La CPU verifica activamente el bit STATUS del controlador en un bucle. Simple pero ineficiente: la CPU desperdicia 100% de ciclos esperando. Solo útil en sistemas embebidos con periférico único.

while(STATUS != READY);
leer/escribir DATA;
// CPU bloqueada todo el tiempo
⚡

2 · E/S por interrupciones

La CPU inicia la operación y regresa a otro proceso. Cuando el periférico termina, levanta una IRQ. La CPU maneja la interrupción, transfiere datos y continúa. El bus sólo se usa cuando hay datos reales.

escribir_comando();
dormir_proceso();
// IRQ → despertar → leer DATA
🚀

3 · DMA — Acceso Directo a Memoria

El controlador DMA transfiere bloques entre dispositivo y RAM sin involucrar a la CPU byte a byte. La CPU sólo configura origen, destino y tamaño. Al terminar el bloque, el DMA genera una sola IRQ. Ideal para discos y redes.

configurar_DMA(src,dst,len);
iniciar_DMA();
// CPU libre → IRQ al final
🗺️

4 · Memory-Mapped I/O (MMIO)

Los registros del controlador se mapean en el espacio de direcciones de memoria. La CPU los lee/escribe con instrucciones normales (MOV, LOAD, STORE). Elimina puertos de E/S dedicados. Usado por GPU, PCI-e, controladores modernos.

volatile uint32_t* reg
= (uint32_t*)0xFEE00000;
*reg = COMANDO; // ← escribe HW
