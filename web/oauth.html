SECCIÓN 01

¿Qué es OAuth 2.0?

Un estándar abierto de autorización que permite a las aplicaciones acceder a recursos de un usuario sin que éste entregue su contraseña.

DEFINICIÓN FORMAL

OAuth 2.0 (Open Authorization) es un protocolo de autorización definido en el RFC 6749 que permite a una aplicación de terceros (cliente) obtener acceso limitado a un servicio web en nombre del usuario, ya sea directamente o por delegación. No es un protocolo de autenticación por sí solo, aunque en la práctica se combina con OpenID Connect (OIDC) para lograrlo.

Piensa en esto con una analogía simple: cuando te hospedas en un hotel, el recepcionista te da una tarjeta llave que solo abre tu habitación, no el resto del edificio. OAuth funciona igual: tu aplicación recibe un token que da acceso solo a los recursos que el usuario autorizó, sin necesidad de conocer la contraseña real.

En el contexto de este curso: cuando tu aplicación permite "Iniciar sesión con Google", tu backend nunca ve la contraseña de Google del usuario. En cambio, Google emite un token que tu app puede usar para obtener el perfil del usuario.

✦ VENTAJAS

  • El usuario nunca comparte su contraseña con tu app
  • Acceso granular: solo los permisos estrictamente necesarios
  • Los tokens pueden revocarse sin cambiar contraseñas
  • Reduce la carga de seguridad en tu backend (no almacenas passwords)
  • Mejora la experiencia de usuario: un clic para autenticarse
  • Estándar abierto, soporte en todos los lenguajes y frameworks
  • Compatible con aplicaciones móviles, SPA y servicios de terceros

✦ DESVENTAJAS

  • Curva de aprendizaje inicial pronunciada
  • Dependencia de un proveedor externo (si Google falla, tú fallas)
  • Configuración inicial más compleja que auth tradicional
  • Los tokens robados son difíciles de invalidar si no se implementa revocación
  • Flujos distintos para web, móvil y server-to-server pueden confundir
  • Usuarios sin cuenta en el proveedor no pueden acceder
SECCIÓN 02

Conceptos esenciales

Antes de implementar, necesitamos manejar el vocabulario del protocolo. Estos son los actores y elementos clave.

ACTOR

Resource Owner (Propietario)

El usuario final. Es quien posee los datos y otorga el permiso de acceso a tu aplicación. Ejemplo: la persona que hace clic en "Iniciar sesión con Google".

ACTOR

Client (Cliente)

Tu aplicación (el backend NestJS). Es quien solicita acceso a los recursos del usuario. Debe registrarse primero en el proveedor.

ACTOR

Authorization Server

El servidor que autentica al usuario y emite los tokens. En este caso es Google. Verifica la identidad y otorga el acceso.

ACTOR

Resource Server

El servidor que aloja los datos protegidos. Puede ser la API de Google (perfil, Gmail, etc.) o tu propio backend con datos del usuario.

TOKEN

Access Token

Token de corta duración (minutos/horas) que da acceso a los recursos. Se envía en el encabezado Authorization: Bearer <token> de cada petición.

TOKEN

Refresh Token

Token de larga duración usado para obtener un nuevo Access Token sin que el usuario vuelva a autenticarse. Se guarda de forma segura en el servidor.

TOKEN

ID Token (OIDC)

Un JWT emitido por OpenID Connect que contiene información del usuario (nombre, email, foto). OAuth puro no lo incluye; OIDC lo añade.

CONCEPTO

Scopes (Alcances)

Permisos específicos que tu app solicita. Ejemplo: openid, email, profile. El usuario ve y aprueba exactamente qué puede acceder tu app.

CONCEPTO

Redirect URI / Callback

La URL de tu backend a la que Google redirige al usuario después de que aprueba el acceso. Debe registrarse en Google Console exactamente.

CONCEPTO

Client ID & Secret

Credenciales que identifican tu aplicación ante Google. El Client ID es público; el Client Secret es confidencial y nunca va al frontend.

CONCEPTO

Authorization Code

Código temporal de un solo uso que Google devuelve al callback. Tu backend lo intercambia por los tokens reales. Dura solo unos minutos.

CONCEPTO

Grant Type

El flujo OAuth a usar. El más común para web es Authorization Code. Otros: Implicit (obsoleto), Client Credentials (server-to-server), Device Code.

SECCIÓN 03

Flujo de autorización

El flujo "Authorization Code" es el más seguro para aplicaciones web con backend. Así funciona paso a paso.

// AUTHORIZATION CODE FLOW — VISIÓN GENERAL
Usuario
browser
→
Tu App
NestJS
→
Google
auth server
→
Callback
/auth/google/callback
→
JWT propio
sesión local

Descripción detallada del flujo

1

El usuario hace clic en "Iniciar sesión con Google"

Tu frontend redirige o hace una petición a tu backend en GET /auth/google. Tu backend (usando Passport.js) construye la URL de autorización de Google e redirige al usuario hacia ella, incluyendo tu client_id, los scopes solicitados y tu redirect_uri.

2

Google muestra la pantalla de consentimiento

El usuario ve exactamente qué permisos solicita tu app (email, perfil, etc.) y decide si aprobar o denegar. Nunca interactúa con tu backend en este paso.

3

Google redirige al Callback con un código

Si el usuario aprueba, Google redirige al navegador hacia tu redirect_uri (ej: http://localhost:3000/auth/google/callback?code=4/0AX4XfWi...). Este code es temporal y de un solo uso.

4

Tu backend intercambia el código por tokens

Passport.js maneja esto automáticamente: hace una petición server-to-server a Google entregando el código, el client_id y el client_secret. Google responde con un access_token, un refresh_token y un id_token (JWT con datos del perfil).

5

Tu backend crea una sesión propia y redirige al frontend

Con el perfil del usuario (nombre, email, foto) puedes crear o actualizar un registro en tu base de datos, generar tu propio JWT de sesión, y redirigir al frontend pasándole ese token para que lo guarde localmente.

ℹ
NOTA IMPORTANTE En producción, nunca pases el JWT por URL (?token=...). Usa cookies HttpOnly o un canal seguro. Para este ejercicio educativo lo haremos por URL para simplificar.
SECCIÓN 04

Configurar Google Developer Console

Antes de escribir código, necesitas registrar tu aplicación en Google Cloud para obtener las credenciales OAuth.

⚠
PREREQUISITO Necesitas una cuenta de Google. El proceso tarda unos 5 minutos. Las credenciales que generes son gratuitas para uso de desarrollo y proyectos con límites razonables.
1

Crear un proyecto en Google Cloud Console

Ve a console.cloud.google.com e inicia sesión con tu cuenta de Google.

  • En la barra superior, haz clic en el selector de proyectos (generalmente dice "My First Project" o similar)
  • Haz clic en Nuevo Proyecto en la esquina superior derecha del modal
  • Asigna un nombre descriptivo, ej: mi-app-oauth-nestjs
  • Haz clic en Crear y espera unos segundos
  • Asegúrate de que el nuevo proyecto quede seleccionado en el selector
2

Configurar la pantalla de consentimiento OAuth

Esta es la pantalla que verán tus usuarios cuando tu app les pida permisos.

  • En el menú lateral izquierdo, navega a APIs y Servicios → Pantalla de consentimiento de OAuth
  • Selecciona el tipo de usuario: elige Externo (permite cualquier cuenta de Google)
  • Haz clic en Crear
  • Rellena los campos obligatorios: Nombre de la app (ej: "Mi App Web"), Email de soporte (tu correo), e Email del desarrollador
  • En la sección de Dominios autorizados, para desarrollo no necesitas agregar nada
  • Haz clic en Guardar y continuar
💡
SCOPES (Alcances) En la siguiente pantalla (Permisos/Scopes), haz clic en Agregar o quitar permisos y añade: openid, email y profile. Guarda y continúa.
  • En la pantalla de Usuarios de prueba (importante mientras estés en modo prueba): haz clic en + Add Users y agrega tu propio correo de Google
  • Finaliza el asistente haciendo clic en Guardar y continuar hasta llegar al resumen
3

Crear las credenciales OAuth 2.0 (Client ID y Secret)

Aquí se generan las llaves que usará tu backend.

  • Ve a APIs y Servicios → Credenciales
  • Haz clic en + Crear credenciales → selecciona ID de cliente de OAuth
  • En Tipo de aplicación, selecciona Aplicación web
  • Ponle un nombre, ej: nestjs-backend
  • En Orígenes de JavaScript autorizados, agrega: http://localhost:3000
  • En URIs de redireccionamiento autorizados, agrega exactamente: http://localhost:3000/auth/google/callback
  • Haz clic en Crear
⚠
¡GUARDA ESTAS CREDENCIALES! Google te mostrará una ventana con el Client ID y el Client Secret. Cópialos y guárdalos en un lugar seguro. El Client Secret no se puede volver a ver. Si lo pierdes, debes crear nuevas credenciales.
4

Verificar la habilitación de la API de Google+

Para acceder al perfil del usuario, la API correcta debe estar habilitada.

  • Ve a APIs y Servicios → Biblioteca
  • Busca "Google People API" o "Google+ API"
  • Haz clic en ella y luego en Habilitar si no está habilitada
  • Opcionalmente, también habilita la Google Identity Toolkit API

Crear la Estrategia de Google (Google Strategy)

La estrategia es el corazón del proceso. Passport.js la usa para manejar el flujo OAuth automáticamente. Crea la carpeta src/auth/ y dentro el archivo:

src/auth/google.strategy.ts
TYPESCRIPT 
import { Injectable } from '@nestjs/common';
import { PassportStrategy } from '@nestjs/passport';
import { Strategy, VerifyCallback } from 'passport-google-oauth20';
import { ConfigService } from '@nestjs/config';

@Injectable()
export class GoogleStrategy extends PassportStrategy(Strategy, 'google') {

  constructor(private configService: ConfigService) {
    super({
      clientID:     configService.get('GOOGLE_CLIENT_ID'),
      clientSecret: configService.get('GOOGLE_CLIENT_SECRET'),
      callbackURL:  configService.get('GOOGLE_CALLBACK_URL'),
      scope: ['email', 'profile'], // qué pedimos a Google
    });
  }

  /**
   * validate() se ejecuta automáticamente cuando Google redirige
   * al callback con el perfil del usuario ya verificado.
   * Lo que retornamos aquí se adjunta a req.user
   */
  async validate(
    accessToken: string,
    refreshToken: string,
    profile: any,
    done: VerifyCallback,
  ): Promise<any> {
    const { name, emails, photos } = profile;

    // Construimos un objeto usuario con los datos relevantes
    const user = {
      email:     emails[0].value,
      firstName: name.givenName,
      lastName:  name.familyName,
      picture:   photos[0].value,
      accessToken,
    };

    done(null, user); // null = sin error, user = datos adjuntados a req.user
  }
}
5

Crear el AuthService (generar el JWT propio)

Después de verificar al usuario con Google, generamos nuestro propio JWT de sesión para usar en llamadas futuras a nuestra API.

src/auth/auth.service.ts
TYPESCRIPT 
import { Injectable } from '@nestjs/common';
import { JwtService } from '@nestjs/jwt';

@Injectable()
export class AuthService {

  constructor(private jwtService: JwtService) {}

  /**
   * Recibe el objeto usuario que retornó GoogleStrategy.validate()
   * y genera un JWT firmado con nuestra propia clave secreta.
   */
  generateToken(user: any) {
    const payload = {
      email:     user.email,
      firstName: user.firstName,
      lastName:  user.lastName,
      picture:   user.picture,
      sub:       user.email, // "subject" del JWT — identificador único
    };

    return {
      access_token: this.jwtService.sign(payload),
    };
  }
}
6

Crear el AuthController (rutas de autenticación)

El controlador expone dos endpoints: el que inicia el flujo OAuth y el callback que maneja la respuesta de Google.

src/auth/auth.controller.ts
TYPESCRIPT 
import { Controller, Get, Req, Res, UseGuards } from '@nestjs/common';
import { AuthGuard } from '@nestjs/passport';
import { AuthService } from './auth.service';

@Controller('auth')
export class AuthController {

  constructor(private authService: AuthService) {}

  // PASO 1 del flujo: el frontend redirige aquí para iniciar auth con Google
  @Get('google')
  @UseGuards(AuthGuard('google'))
  async googleAuth() {
    // Passport intercepta esta ruta y redirige automáticamente a Google.
    // El cuerpo de esta función nunca se ejecuta.
  }

  // PASO 3 del flujo: Google redirige aquí con el código de autorización
  @Get('google/callback')
  @UseGuards(AuthGuard('google'))
  async googleCallback(@Req() req: any, @Res() res: any) {
    // req.user contiene lo que retornó GoogleStrategy.validate()
    // Generamos nuestro propio JWT de sesión
    const tokenData = this.authService.generateToken(req.user);

    // Redirigimos al frontend con el token en la URL
    // En producción: usa cookies HttpOnly en lugar de URL params
    res.redirect(
      `http://localhost:5500/index.html?token=${tokenData.access_token}`
    );
  }

  // Endpoint para obtener el perfil del usuario autenticado
  @Get('profile')
  @UseGuards(AuthGuard('jwt'))
  getProfile(@Req() req: any) {
    return req.user; // retorna el payload del JWT
  }
}
ℹ
¿QUÉ ES @UseGuards(AuthGuard('google'))? Es un decorador de NestJS que activa una guardia de Passport. Con 'google' activa nuestra GoogleStrategy. Con 'jwt' activará la estrategia JWT que crearemos a continuación.
7

Crear la estrategia JWT para proteger rutas privadas

Una vez que el usuario tiene su JWT propio, necesitamos una estrategia que valide ese token en las rutas protegidas.

src/auth/jwt.strategy.ts
TYPESCRIPT 
import { Injectable } from '@nestjs/common';
import { PassportStrategy } from '@nestjs/passport';
import { ExtractJwt, Strategy } from 'passport-jwt';
import { ConfigService } from '@nestjs/config';

@Injectable()
export class JwtStrategy extends PassportStrategy(Strategy, 'jwt') {

  constructor(private configService: ConfigService) {
    super({
      // Extrae el token del encabezado Authorization: Bearer <token>
      jwtFromRequest: ExtractJwt.fromAuthHeaderAsBearerToken(),
      ignoreExpiration: false,
      secretOrKey: configService.get('JWT_SECRET'),
    });
  }

  async validate(payload: any) {
    // payload es el contenido decodificado del JWT
    // Lo que retornamos se adjunta a req.user en las rutas protegidas
    return {
      email:     payload.email,
      firstName: payload.firstName,
      lastName:  payload.lastName,
      picture:   payload.picture,
    };
  }
}
8

Ensamblar el AuthModule

El módulo une todas las piezas: estrategias, servicio, controlador y configuración de JWT.

src/auth/auth.module.ts
TYPESCRIPT 
import { Module } from '@nestjs/common';
import { PassportModule } from '@nestjs/passport';
import { JwtModule } from '@nestjs/jwt';
import { ConfigModule, ConfigService } from '@nestjs/config';

import { AuthController } from './auth.controller';
import { AuthService } from './auth.service';
import { GoogleStrategy } from './google.strategy';
import { JwtStrategy } from './jwt.strategy';

@Module({
  imports: [
    PassportModule,
    JwtModule.registerAsync({
      imports: [ConfigModule],
      useFactory: (configService: ConfigService) => ({
        secret:      configService.get('JWT_SECRET'),
        signOptions: { expiresIn: '7d' }, // el token expira en 7 días
      }),
      inject: [ConfigService],
    }),
  ],
  controllers: [AuthController],
  providers: [AuthService, GoogleStrategy, JwtStrategy],
})
export class AuthModule {}

Habilitar CORS en main.ts

Para que el frontend HTML pueda comunicarse con el backend, debemos habilitar CORS.

src/main.ts
TYPESCRIPT 
import { NestFactory } from '@nestjs/core';
import { AppModule } from './app.module';

async function bootstrap() {
  const app = await NestFactory.create(AppModule);

  app.enableCors({
    origin: ['http://localhost:5500', 'http://127.0.0.1:5500'],
    methods: ['GET', 'POST', 'PUT', 'DELETE'],

  });

const API_URL = 'http://localhost:3000';

// ─── Al cargar la página ───────────────────────────────────────────
function init() {
  const params = new URLSearchParams(window.location.search);
  const tokenFromCallback = params.get('token');

  if (tokenFromCallback) {
    // Venimos del callback de Google — guardamos el token
    localStorage.setItem('auth_token', tokenFromCallback);
    // Limpiamos el token de la URL por seguridad
    window.history.replaceState({}, document.title, '/');
  }

  const savedToken = localStorage.getItem('auth_token');

  if (savedToken) {
    showProfile(savedToken);
  } else {
    showLoginView();
  }
}

// ─── Decodificar el JWT y mostrar el perfil ────────────────────────
function showProfile(token) {
  try {
    // Los JWT tienen 3 partes separadas por '.': header.payload.signature
    // Decodificamos el payload (parte central) en Base64
    const payload = JSON.parse(atob(token.split('.')[1]));

    document.getElementById('user-avatar').src = payload.picture || '';
    document.getElementById('user-name').textContent =
      `${payload.firstName} ${payload.lastName}`;
    document.getElementById('user-email').textContent = payload.email;

    document.getElementById('login-view').style.display = 'none';
    document.getElementById('profile-view').style.display = 'block';
  } catch (e) {
    logout(); // token inválido → volvemos al login
  }
}

function showLoginView() {
  document.getElementById('login-view').style.display = 'block';
  document.getElementById('profile-view').style.display = 'none';
}

// ─── Llamar a una ruta protegida del backend ───────────────────────
async function callProtectedAPI() {
  const token = localStorage.getItem('auth_token');
  const resultEl = document.getElementById('api-result');

  try {
    const res = await fetch(`${API_URL}/auth/profile`, {
      headers: {
        // Enviamos el JWT en el encabezado Authorization
        'Authorization': `Bearer ${token}`
      }
    });

    if (res.ok) {
      const data = await res.json();
      resultEl.textContent = JSON.stringify(data, null, 2);
      resultEl.style.display = 'block';
    } else {
      resultEl.textContent = 'Error: token inválido o expirado';
      resultEl.style.display = 'block';
    }
  } catch (e) {
    resultEl.textContent = 'Error de red: ¿está corriendo el backend?';
    resultEl.style.display = 'block';
  }
}

// ─── Cerrar sesión ─────────────────────────────────────────────────
function logout() {
  localStorage.removeItem('auth_token');
  showLoginView();
  document.getElementById('api-result').style.display = 'none';
}

// Ejecutar al cargar
init();
</script>

Probar el flujo completo

  • Inicia el backend: npm run start:dev en la carpeta de NestJS
  • Abre el index.html con Live Server (VS Code) → aparece en http://localhost:5500
  • Haz clic en "Continuar con Google"
  • Selecciona tu cuenta de Google (la que agregaste como usuario de prueba)
  • Aprueba los permisos solicitados
  • Google redirige a tu callback → tu backend genera el JWT → redirige al frontend
  • El frontend captura el token de la URL, lo guarda en localStorage y muestra tu perfil
  • Haz clic en "Llamar ruta protegida" para ver la respuesta del backend con tu perfil decodificado del JWT
RESUMEN

Lo que construimos

BACKEND

GoogleStrategy

Maneja el flujo OAuth con Google usando Passport.js. Extrae el perfil del usuario y lo pasa al controlador.

BACKEND

JwtStrategy

Valida el JWT propio en cada petición a rutas protegidas. Inyecta el perfil en req.user.

BACKEND

AuthService

Genera un JWT firmado con datos del usuario de Google. Este token es el que usamos en nuestra app.

FRONTEND

index.html

Detecta el token en la URL al volver del callback, lo guarda en localStorage y lo envía en el header Authorization.
