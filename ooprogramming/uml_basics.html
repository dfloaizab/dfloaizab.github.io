Principios del Modelado Orientado a Objetos

El modelado orientado a objetos (MOO) es un enfoque para analizar y diseñar sistemas de software representando el problema mediante objetos que interactúan entre sí. Sus principios guían la construcción de modelos robustos, mantenibles y comprensibles.

🔵 Abstracción

Representar solo las características esenciales de una entidad, ignorando detalles irrelevantes para el contexto.

🔒 Encapsulamiento

Ocultar el estado interno de un objeto y exponer solo una interfaz controlada de acceso.

🧬 Herencia

Permitir que una clase reutilice atributos y métodos de otra, promoviendo la reutilización del código.

🔀 Polimorfismo

Permitir que distintos objetos respondan de forma diferente a una misma operación o mensaje.

🔗 Modularidad

Dividir el sistema en componentes independientes que puedan desarrollarse y probarse por separado.

🏗️ Jerarquía

Organizar las clases en estructuras jerárquicas que reflejen relaciones de generalización y especialización.

¿Qué es UML?

El Lenguaje Unificado de Modelado (UML, por sus siglas en inglés: Unified Modeling Language) es un lenguaje visual estandarizado para especificar, visualizar, construir y documentar los artefactos de un sistema de software. Fue adoptado por el Object Management Group (OMG) como estándar en 1997 y actualmente se encuentra en su versión 2.x.

UML no es un lenguaje de programación ni una metodología de desarrollo. Es un lenguaje de modelado: una notación gráfica y textual que permite comunicar decisiones de diseño entre los miembros de un equipo, independientemente del lenguaje de programación que se vaya a utilizar.

Sus creadores originales fueron Grady Booch, Ivar Jacobson y James Rumbaugh, conocidos como "los tres amigos", quienes unificaron sus metodologías de diseño orientado a objetos en un único lenguaje durante los años 90.

Características de UML

Estándar

Notación universal

UML es reconocido internacionalmente como el estándar para modelado de software orientado a objetos, lo que facilita la comunicación entre equipos de diferentes países y empresas.

Visual

Representación gráfica

Utiliza diagramas con íconos, cajas y flechas estandarizados para representar clases, objetos, relaciones, flujos y comportamientos de manera intuitiva.

Independiente

Independiente del lenguaje

Los modelos UML pueden implementarse en cualquier lenguaje orientado a objetos: Java, C++, Python, C#, entre otros, sin modificar el diagrama.

Escalable

Aplicable a cualquier escala

Puede modelar desde una pequeña clase hasta arquitecturas empresariales complejas con múltiples subsistemas y miles de componentes.

Completo

Modela estructura y comportamiento

No solo describe la estructura estática del sistema (clases, atributos) sino también su comportamiento dinámico (secuencias, estados, actividades).

Formal

Semántica bien definida

Cada elemento del lenguaje tiene una semántica precisa, lo que reduce la ambigüedad en la interpretación de los modelos por parte del equipo de desarrollo.

Principales tipos de diagramas UML

UML 2.x define 14 tipos de diagramas agrupados en dos categorías:

Diagramas estructurales

Diagrama de clases — el más utilizado. Muestra clases, atributos, métodos y sus relaciones.

Diagrama de objetos — muestra instancias concretas de clases en un momento dado.

Diagrama de componentes — organización de los componentes físicos del sistema.

Diagrama de paquetes — agrupa clases en paquetes lógicos.

Diagrama de despliegue — distribución del sistema en hardware.

Diagramas de comportamiento

Diagrama de casos de uso — interacción entre actores y el sistema.

Diagrama de secuencia — mensajes entre objetos en el tiempo.

Diagrama de estados — ciclo de vida de un objeto.

Diagrama de actividades — flujo de control y trabajo.

Diagrama de comunicación — colaboración entre objetos.

Tipos de relaciones en UML

Las relaciones entre clases son el núcleo del diagrama de clases. UML define los siguientes tipos fundamentales:

Relación Notación UML Significado Ejemplo real
Asociación Línea sólida con flecha → Un objeto conoce o usa a otro. Relación general entre clases. Estudiante → Universidad
Herencia (Generalización) Línea sólida con triángulo vacío △ Una clase hija extiende una clase padre. Relación "es un". Perro △ Animal
Realización (Implementación) Línea punteada con triángulo vacío △ Una clase implementa los métodos definidos en una interfaz. Circulo implements Figura
Dependencia Línea punteada con flecha --→ Un objeto usa temporalmente a otro (parámetros o variables locales). Pedido --→ Descuento
Agregación Línea sólida con rombo vacío ◇ Un objeto contiene a otros, pero estos pueden existir de forma independiente. Relación "tiene un". Universidad ◇ Departamento
Composición Línea sólida con rombo relleno ◆ Un objeto contiene a otros y éstos no pueden existir sin él. Relación "está compuesto de". Casa ◆ Habitación

Ejemplos prácticos con código Java

Ejemplo 1 — Asociación: Estudiante y Universidad

Un Estudiante conoce a una Universidad. El estudiante tiene una referencia al objeto universidad, pero ambos pueden existir de forma independiente. Esta es la relación más básica entre clases. 
class Universidad {
    String nombre;
    String ciudad;

    Universidad(String nombre, String ciudad) {
        this.nombre = nombre;
        this.ciudad = ciudad;
    }
}

class Estudiante {
    String nombre;
    Universidad universidad; // asociación

    Estudiante(String nombre, Universidad universidad) {
        this.nombre = nombre;
        this.universidad = universidad;
    }

    void presentarse() {
        System.out.println("Soy " + nombre +
            " y estudio en " + universidad.nombre);
    }
}

// Uso:
// Universidad u = new Universidad("UNIVALLE", "Cali");
// Estudiante e = new Estudiante("Ana", u);
// e.presentarse();

Ejemplo 2 — Herencia: Animal, Perro y Gato

Perro y Gato son tipos específicos de Animal. Heredan el método respirar() y sobrescriben hacerSonido() con su propio comportamiento (polimorfismo). 
class Animal {
    String nombre;

    Animal(String nombre) {
        this.nombre = nombre;
    }

    void respirar() {
        System.out.println(nombre + " está respirando");
    }

    void hacerSonido() {
        System.out.println("Sonido genérico");
    }
}

class Perro extends Animal {
    Perro(String nombre) { super(nombre); }

    @Override
    void hacerSonido() {
        System.out.println(nombre + " dice: ¡Guau!");
    }
}

class Gato extends Animal {
    Gato(String nombre) { super(nombre); }

    @Override
    void hacerSonido() {
        System.out.println(nombre + " dice: ¡Miau!");
    }
}

// Uso (polimorfismo):
// Animal[] animales = { new Perro("Rex"), new Gato("Luna") };
// for (Animal a : animales) a.hacerSonido();

Ejemplo 3 — Realización: Interfaz Figura

La interfaz Figura define un contrato con el método abstracto calcularArea(). Las clases Circulo y Rectangulo realizan ese contrato implementando el método con su propia fórmula. 
interface Figura {
    double calcularArea();
    double calcularPerimetro();
}

class Circulo implements Figura {
    double radio;

    Circulo(double radio) { this.radio = radio; }

    @Override
    public double calcularArea() {
        return Math.PI * radio * radio;
    }

    @Override
    public double calcularPerimetro() {
        return 2 * Math.PI * radio;
    }
}

class Rectangulo implements Figura {
    double base, altura;

    Rectangulo(double base, double altura) {
        this.base = base;
        this.altura = altura;
    }

    @Override
    public double calcularArea() { return base * altura; }

    @Override
    public double calcularPerimetro() { return 2 * (base + altura); }
}

// Uso:
// Figura f1 = new Circulo(5);
// Figura f2 = new Rectangulo(4, 6);
// System.out.println(f1.calcularArea());   // 78.53...
// System.out.println(f2.calcularArea());   // 24.0

Ejemplo 4 — Agregación: Universidad y Departamento

Una Universidad tiene varios Departamentos. Si la universidad cerrara, los departamentos podrían reorganizarse en otra institución: sus objetos existen de forma independiente. Esto caracteriza la agregación (rombo vacío ◇). 
import java.util.ArrayList;
import java.util.List;

class Departamento {
    String nombre;
    String jefe;

    Departamento(String nombre, String jefe) {
        this.nombre = nombre;
        this.jefe   = jefe;
    }

    void mostrarInfo() {
        System.out.println("Depto: " + nombre + " | Jefe: " + jefe);
    }
}

class Universidad {
    String nombre;
    List<Departamento> departamentos; // agregación

    Universidad(String nombre) {
        this.nombre        = nombre;
        this.departamentos = new ArrayList<>();
    }

    void agregarDepartamento(Departamento d) {
        departamentos.add(d);
    }

    void listarDepartamentos() {
        System.out.println("Departamentos de " + nombre + ":");
        for (Departamento d : departamentos) d.mostrarInfo();
    }
}

// Uso:
// Departamento d1 = new Departamento("Sistemas", "Dr. Pérez");
// Departamento d2 = new Departamento("Matemáticas", "Dra. López");
// Universidad u = new Universidad("UNIVALLE");
// u.agregarDepartamento(d1);
// u.agregarDepartamento(d2);
// u.listarDepartamentos();

Ejemplo 5 — Composición: Casa y Habitaciones

Una Casa está compuesta de Habitaciones. Si la casa es destruida, sus habitaciones dejan de existir con ella. Esto caracteriza la composición (rombo relleno ◆): el ciclo de vida de las partes depende del todo. 
import java.util.ArrayList;
import java.util.List;

class Habitacion {
    String tipo;     // "Sala", "Cocina", "Dormitorio", etc.
    double metrosCuadrados;

    // El constructor es privado de paquete: solo Casa puede crear habitaciones
    Habitacion(String tipo, double metrosCuadrados) {
        this.tipo             = tipo;
        this.metrosCuadrados  = metrosCuadrados;
    }

    void describir() {
        System.out.println(tipo + " (" + metrosCuadrados + " m²)");
    }
}

class Casa {
    String direccion;
    private List<Habitacion> habitaciones; // composición

    Casa(String direccion) {
        this.direccion   = direccion;
        this.habitaciones = new ArrayList<>();
        // Las habitaciones se crean dentro de la casa
        habitaciones.add(new Habitacion("Sala",       25.0));
        habitaciones.add(new Habitacion("Cocina",     15.0));
        habitaciones.add(new Habitacion("Dormitorio", 20.0));
    }

    void mostrarPlano() {
        System.out.println("Casa en: " + direccion);
        for (Habitacion h : habitaciones) h.describir();
    }
}

// Uso:
// Casa c = new Casa("Calle 5 # 12-34, Cali");
// c.mostrarPlano();
